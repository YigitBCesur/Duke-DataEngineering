### 1. Kurulum ve Başlangıç
Bu komutlar bir projeye başlarken sadece bir kez kullanılır.
* **`git init`**: Bulunduğun klasörü bir Git deposu (repository) yapar. `.git` klasörünü oluşturur.
* **`git config --global user.name "Adın"`**: Git'e kim olduğunu tanıtır.

### 2. Gündelik Çalışma Akışı (En Çok Kullanacakların)
Dosyalar üzerinde değişiklik yaptıkça bu döngüyü sürekli kullanacaksın.
* **`git status`**: En yakın dostundur. Hangi dosyalar değişti, hangileri "sahneye" eklendi, terminalde kırmızı/yeşil renklerle gösterir.
* **`git add <dosya_adi>`**: Belirli bir dosyayı bir sonraki kayıt (commit) için hazırlar.
* **`git add .`**: Klasördeki **tüm** değişiklikleri hazırlık alanına (staging area) ekler.
* **`git commit -m "Mesajın"`**: Hazırladığın değişiklikleri kalıcı olarak bir "versiyon" olarak kaydeder. Mesaj kısmına ne yaptığını net yazmak (Örn: "Hata düzeltildi") ileride hayat kurtarır.



### 3. İnceleme ve Takip
Neler olup bittiğini anlamak için kullanılır.
* **`git diff`**: Henüz `add` yapmadığın değişiklikleri satır satır gösterir (Az önce yeşil çizgilerle gördüğün ekran).
* **`git log`**: Yapılan tüm kayıtların (commit) geçmişini listeler. Kim, ne zaman, ne yazmış?
* **`git log --oneline`**: Geçmişi çok daha kısa ve öz bir liste olarak sunar.

### 4. Hata Yönetimi ve Geri Dönüş
İşler karıştığında kullanılan "acil durum" komutları.
* **`git checkout <dosya_adi>`**: Dosyadaki henüz kaydedilmemiş (commit edilmemiş) değişiklikleri siler ve dosyayı son kayıtlı haline geri döndürür.
* **`git reset HEAD <dosya_adi>`**: Yanlışlıkla `add` yaptığın bir dosyayı hazırlık alanından geri çıkarır (dosya içeriği değişmez).

### 5. Git stash
git stash, Git'in en hayat kurtaran "geçici hafıza" özelliğidir.

Şöyle bir senaryo düşün: git_example.py üzerinde derin bir kod değişikliği yapıyorsun ama işin henüz bitmedi (kod şu an çalışmıyor). Tam o sırada çok acil bir hata çıktı ve başka bir dala (branch) geçmen ya da tertemiz bir kodla bir şeyi test etmen gerekiyor.

Yarım kalan kodunu commit etmek istemezsin (çünkü bozuk), ama silmek de istemezsin. İşte burada git stash devreye girer.

git stash Nasıl Çalışır?
Bu komutu yazdığında Git, henüz commit etmediğin tüm değişiklikleri alır ve güvenli bir "rafa" kaldırır. Çalışma dizinin (working directory) tertemiz, yani son commit yaptığın haline döner.

En Çok Kullanılan Stash Komutları
git stash: Değişiklikleri rafa kaldırır ve ortalığı temizler.

git stash list: Rafta bekleyen tüm "paketlerini" listeler. (Her stash'in bir indeksi vardır, stash@{0} gibi).

git stash pop: Raftaki en son değişikliği geri getirir ve raftan siler. (Sanki hiç gitmemiş gibi kodun geri gelir).

git stash apply: Değişikliği geri getirir ama rafta bir kopyasını bırakmaya devam eder.

git stash drop: Raftaki belirli bir değişikliği kalıcı olarak siler.

git stash clear: Raftaki her şeyi temizler.

### 6. Git branch

Git'in en güçlü özelliklerinden biri olan Branch (Dal) mantığı, projenin ana hattını bozmadan farklı özellikler veya denemeler üzerinde çalışmanı sağlar.

Az önce git stash ile kodlarını rafa kaldırmayı öğrendin; branch ise projeni paralel evrenlere bölmek gibidir.

1. Neden Branch Kullanılır?
Diyelim ki git_example.py üzerinde yeni bir özellik ekleyeceksin ama bu özellik ana kodu bozabilir. "Master" (veya "Main") dalından yeni bir dal oluşturursun, orada istediğin kadar hata yapar, düzeltir ve işin bitince ana dalla birleştirirsin.

2. En Temel Branch Komutları
git branch: Mevcut tüm dalları listeler. Yanında yıldız (*) olan, şu an üzerinde bulunduğun daldır.

git branch <new-branch-adi>: Yeni bir dal oluşturur ama seni o dala geçirmez.

git checkout <brach-adi> veya git switch <branch-adi>: Başka bir dala geçiş yapmanı sağlar.

git checkout -b <new-branch-adi>: Hem yeni bir dal oluşturur hem de seni anında o dala geçirir (En çok bu kullanılır).

3. Branch(Dal)ları Birleştirme (Merge)
Yeni daldaki işin bitti ve her şey harika çalışıyor. Bu değişiklikleri ana dala aktarmak için:

Önce ana dala geç: git checkout main

Diğer dalı buraya çek: git merge <new-branch-adi>


### 7. Git log
git log komutu, projenin zaman makinesi gibidir. Bu komutu yazdığında, o ana kadar yaptığın tüm "commit"leri (kayıt duraklarını) kronolojik olarak listelersin.
1. git log Çıktısında Neler Görürsün?
Komutu yazdığında karşına çıkan her blok şunları içerir:

Commit Hash: commit a1b2c3d... şeklinde uzun bir kod. Bu, o kaydın benzersiz kimliğidir.

Author: Değişikliği kimin yaptığı (Senin ismin ve e-postan).

Date: Kaydın yapıldığı tam zaman.

Message: Senin git commit -m ile yazdığın açıklama.

2. Hayat Kurtaran git log Parametreleri
Standart git log bazen çok uzun ve okunması zor olabilir. Veri mühendisliği projelerinde şu versiyonları daha sık kullanırız:

Tek Satırda Özet: (En çok kullanılan budur)

Bash
git log --oneline
Grafiksel Görünüm: (Terminalde dallanmaları görmek için)

Bash
git log --oneline --graph --all
Son N Kaydı Gör: (Sadece son 3 kaydı görmek için)

Bash
git log -n 3
Dosya Bazlı Takip: (Sadece hello.py dosyasındaki değişimleri görmek için)

Bash
git log hello.py
3. Log'dan Nasıl Çıkılır?
Eğer log listen çok uzunsa, terminalin altında : işaretini görürsün ve ekran donmuş gibi gelebilir.

Çıkmak için: Klavyeden sadece q (quit) tuşuna basman yeterlidir.

Senin İçin Bir İpucu:
Daha önce yaptığın o "git stash added to file" veya "updated" mesajlı commit'lerini şu an terminalde git log --oneline yazarak görmeyi dene. En üstteki commit'in yanında muhtemelen (HEAD -> master) veya (HEAD -> deneme) yazacaktır.



### Küçük Bir Tavsiye
Terminalde çalışırken `git status` yazmayı bir alışkanlık (refleks) haline getir. Her `add` ve her `commit` öncesinde durumu kontrol etmek, yanlış dosyayı kaydetmeni önler.

Şimdi bu komutlarla kendi `git_example.py` dosyan üzerinde biraz daha "oynayabilirsin". Mesela bir satır silip `git diff` ile nasıl göründüğüne bakmaya ne dersin?
