🎓 Ders: Dağıtılmış Sürüm Kontrolü ve GitHub Dünyası
Bu ders, kodlarınızın sadece bilgisayarınızda hapsolmamasını, bulut üzerinde güvenle saklanmasını ve başkalarıyla iş birliği içinde geliştirilmesini sağlar.

1. Temel Kavram: Dağıtılmış Sürüm Kontrolü (DVCS)
Geleneksel sistemlerin aksine Git, "Dağıtılmış" bir sistemdir.

Tam Kopya: Projeyi "clone" ettiğinizde (kopyaladığınızda), sadece dosyaları değil, projenin tüm geçmişini (tüm commit'leri) bilgisayarınıza indirirsiniz.

Bağımsızlık: İnternetiniz olmasa bile kendi bilgisayarınızda (Local) çalışabilir, commit yapabilir ve dallar (branch) oluşturabilirsiniz. İnternet geldiğinde bu değişiklikleri ana sunucuya (GitHub) gönderebilirsiniz.

2. Yerel (Local) vs. Uzak (Remote) Repo
Veri mühendisliği projelerinde iki farklı çalışma alanınız vardır:

Local (Yerel): MSI laptopun içindeki WSL/Ubuntu terminali. Burada git add ve git commit komutlarını kullanırsın.

Remote (Uzak): GitHub.com üzerindeki sunucular. Burada verilerin dünya genelinde erişilebilir ve yedekli haldedir.

3. GitHub'da Yeni Bir Hayat: Depo (Repository) Oluşturma
GitHub üzerinde bir proje başlatırken karşınıza çıkan kritik terimler:

Public (Açık): Kodun herkes tarafından görülebilir. Portfolyo oluşturmak için idealdir.

Private (Gizli): Sadece senin (ve izin verdiğin kişilerin) görebileceği depolardır.

README.md: Projenin "vitrinidir". Projenin ne yaptığını, nasıl kurulacağını buraya Markdown (.md) formatında yazarsın. (Hatırlarsan .docx yerine bunu tercih ediyorduk!)

.gitignore: Git'e "Şu dosyaları sakın takip etme" dediğin listedir. (Örn: Dev veri dosyaları veya şifreler).

4. Terminal ve GitHub Arasındaki Köprü: Temel Komutlar
A. Projeyi Bilgisayara İndirme
GitHub'da oluşturduğun bir projeyi bilgisayarına çekmek için:

Bash
git clone https://github.com/kullanici_adin/proje_adin.git
B. Değişiklikleri Yukarı Gönderme (PUSH)
Kendi bilgisayarında bir dosyayı değiştirdin, add yaptın ve commitledin. Şimdi bunu GitHub'a gönderme vakti:

Bash
git push origin master
(Buradaki origin GitHub'ı, master ise ana dalı temsil eder.)

C. Değişiklikleri Bilgisayara Çekme (PULL)
Eğer GitHub arayüzünden bir dosyayı düzenlediysen veya bir iş arkadaşın değişiklik yaptıysa, o değişiklikleri kendi bilgisayarına almak için:

Bash
git pull origin master
5. Terminal Editörleri Notu: Nano vs. Vim
Dersimizin başında konuştuğumuz gibi, terminalde hızlıca bir dosya (örneğin README) düzenlemek istersen:

Nano: Hızlı ve kolay. nano README.md yaz, düzenle, Ctrl+O ile kaydet.

Vim: Profesyonel ve hızlı. vim README.md yaz, i ile yazmaya başla, çıkmak için Esc ve :wq kullan.

6. Veri Mühendisleri İçin Altın Kurallar
Format Seçimi: Verilerin için .xlsx yerine .csv, dökümanların için .docx yerine .md kullan. Böylece git log ve git diff ile her değişikliği görebilirsin.

Mesaj Kalitesi: Commit mesajların anlamlı olsun. git commit -m "güncelleme" yerine git commit -m "Vim rehberi eklendi ve Markdown formatına çevrildi" yaz.

Sık Sık Push Yap: Kodlarını sadece bilgisayarında tutma. Gün sonunda mutlaka GitHub'a push yap ki emeğin güvende olsun.
